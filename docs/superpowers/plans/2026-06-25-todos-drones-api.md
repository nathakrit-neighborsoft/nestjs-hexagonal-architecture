# Todos & Drones API Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a `drones` resource and a user-scoped `todos` resource (with optional `droneId` FK) using the existing hexagonal NestJS layout, mirroring the structure of the `expenses` module and the schema shape of the reference repo `nathakrit-neighborsoft/elysia-remote-dts-back-end`.

**Architecture:** Two sibling feature modules â€” `src/drones/` and `src/todos/` â€” each with `adapters/inbounds` (controller + DTOs + .http), `adapters/outbounds` (Drizzle repository), `applications/domains` (branded types + interface + class), `applications/ports` (interface + DI token + query/return types), and `applications/usecases` (one use case per route + `.spec.ts`). Drizzle schemas live in `src/databases/schemas/`; a single new migration `0001_*.sql` is generated by `drizzle-kit generate`. No git operations in this plan (workspace is not a git repo).

**Tech Stack:** NestJS 11, Drizzle ORM 0.42 + drizzle-orm/node-postgres, drizzle-kit 0.30, @nestjs-cls/transactional, builder-pattern, class-validator, class-transformer, vitest, vitest-mock-extended, @faker-js/faker. No new dependencies.

## Assumptions & Risks

- **Assumed:** Project is **not** a git repo â€” no `git add/commit`. Plan does not include version control steps.
- **Assumed:** Existing `expenses` pattern (branded types, `Builder/StrictBuilder`, `mock<Repository>()` from `vitest-mock-extended`, `@faker-js/faker`, AAA test style, `NotFoundException`, `@UseGuards(JwtAuthGuard)`, `@ApiBearerAuth(accessKeyToken)`, `@Transactional()`, `TransactionHost<TransactionalAdapterDrizzleOrm<DrizzleDB>>`) is the canonical reference for both new modules. Deviating from it is out of scope.
- **Assumed:** `drizzle-kit generate` is runnable in this environment and writes a `0001_*.sql` + `meta/0001_snapshot.json` + updates `meta/_journal.json`. If it cannot run, the plan records a manual migration scaffold as a fallback.
- **Assumed:** Decimal values on drones are kept as `string` from the DB layer (matches the expense module's `amount` handling) so we avoid `numeric â†’ number` precision loss. Domain layer returns them as `number` for consumers.
- **Risk:** Misaligned `DrizzleDB` generic types after adding new schemas. Mitigation: each new repository re-declares `type DrizzleDB = NodePgDatabase<typeof schema>;` and imports `* as schema from 'src/databases/schemas'`.
- **Risk:** `drizzle-kit generate` is not runnable (no DB connection / no `pnpm install` is performed in this plan). Mitigation: scaffold the `0001_*.sql` migration by hand following the same shape as `0000_soft_roxanne_simpson.sql` plus a new `todos` table with FKs, and update `meta/_journal.json` + add `meta/0001_snapshot.json` stub.
- **Risk:** Drone `description`/`notes`-style field creep. Mitigation: schema is fixed to the 9 fields the reference repo defines; no extras.

## Impact

- New: 2 Drizzle schemas, 2 NestJS modules, ~5 drones + ~6 todos use cases + their specs, 2 controllers, 4 DTOs, 2 http files, 1 migration.
- Modified: `src/databases/schemas/index.ts`, `src/app.module.ts`.
- Untouched: `expenses`, `users`, `auth`, `health`, `configs`, `types`, `utils`, `databases/database.module.ts`, `databases/drizzle.database.providers.ts`.

## Out of Scope (ponytail/YAGNI)

- Pagination / search / sort on `todos` (personal task list â€” keep it simple).
- Soft delete on either table.
- Auth on `GET /drones` (matches reference repo).
- `drone` aggregate endpoints (no `/drones/:id/todos` join endpoint â€” client can filter `GET /todos?droneId=`).
- Drones seeding script.
- Translation/i18n on Swagger descriptions.
- Reorganising `databases/schemas/` directory layout.

    userIdIdx: index('IDX_TODOS_USER_ID').on(table.userId),
